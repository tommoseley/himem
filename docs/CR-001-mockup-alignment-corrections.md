## 2. Correction Scope

### 2.1 Data Model Corrections (TA-001)

#### Add: `Topic` Entity

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| id | UUID | Yes | Unique identifier |
| name | String | Yes | Display name (e.g., "Garden", "Astro") |
| slug | String | Yes | Normalized key for filtering |
| inferred_at | Date | Yes | When the topic was first inferred |
| entry_count | Int | No | Cached count of associated entries |

- Many-to-many with JournalEntry (an entry can belong to multiple topics)
- Topics are system-inferred, not user-created

#### Add: `InferenceSummary` Entity

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| id | UUID | Yes | Unique identifier |
| entry_id | UUID | Yes | Reference to source journal entry |
| summary_text | String | Yes | Natural-language inference description |
| feedback_state | String | No | confirmed, edited, ignored, null (pending) |
| feedback_at | Date | No | When user responded |
| user_correction | String | No | User-provided correction text (if edited) |
| created_at | Date | Yes | Generation timestamp |

- One-to-one with JournalEntry
- Generated by AI Processing Engine alongside entity extraction

#### Revise: `JournalEntry` Entity

| Change | Field | Type | Description |
|--------|-------|------|-------------|
| Add | title | String (optional) | Inferred or derived entry title ("Hands-free capture", "Garden session") |
| Clarify | input_type | String | Must distinguish: `siri`, `voice_in_app`, `typed` (mockup differentiates "Captured via Siri" from "Typed-in app") |

#### Revise: `ProcessingTask` Entity

| Change | Field | Type | Description |
|--------|-------|------|-------------|
| Add | progress_description | String (optional) | Human-readable processing status ("Raw note saved. The app is extracting beds, plant condition, and content intent.") |

### 2.2 Component Corrections (TA-001)

#### Add: Topic Inference Engine

- Derives topic/category assignments from extracted entities and entry content
- Runs as a post-processing step after entity extraction
- Produces `Topic` associations for each entry
- Location: `src/ai/topics/`

#### Add: Inference Summary Generator

- Produces natural-language summaries of AI conclusions for each entry
- Runs as part of AI Processing Engine output
- Generates the "App is Inferring" card content
- Location: `src/ai/summary/`

#### Add: Feedback Controller

- Handles user responses to inference cards (confirm/edit/ignore)
- Persists feedback state on `InferenceSummary`
- Location: `src/ui/feedback/`

#### Add: Unified Input Bar

- Composes text input and voice/Siri readiness into a single persistent bottom bar
- Manages input mode switching (text entry vs. Siri-ready state)
- Coordinates with Voice Input Controller and Text Input Controller
- Location: `src/ui/input/`

#### Revise: Journal Display Controller

- Entry cards must render the full structure shown in mockup: title, source badge, status badge, content, processing card, inline tags, inference summary card, feedback controls
- Must observe processing status changes and update cards reactively
- Must include Siri shortcut banner element

- Entry title (inferred)

## 3. Duplicate Finding Reassessment

EVAL-001 in the binder flagged WS-138 / WS-145 as potential duplicates (62% scope overlap). This remains valid. Both create Core Data storage managers for different entity types. The correction request does not alter this finding, but the addition of `Topic` and `InferenceSummary` models means the data layer is now more complex and the storage architecture should be reviewed holistically.

---

## 4. Execution Order Impact

The original execution order was strictly sequential: WP-025 -> WP-026 -> WP-027.

With the corrections, there are cross-WP dependencies:
- WS-139 (WP-025) now has slots that depend on WP-026 and WP-027 components
- Topic tabs (WP-027) depend on topic inference (WP-026)
- Inference summary display (WP-027) depends on summary generation (WP-026)

**Recommended approach:** Split WS-139 into early (card structure) and late (enrichment) phases. The early phase remains in WP-025's execution order. The late phase becomes a WP-027 concern.
